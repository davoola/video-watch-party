# 私密影院 (PrivateCinema)

一个只为两个人准备的私密在线观影网站：登录后自动扫描本地视频目录（支持多级文件夹浏览），双方可以同步观看同一部视频（播放/暂停/拖进度条都会实时同步），并支持实时聊天——支持 Markdown 格式、表情、图片（点击可全屏 lightbox 查看，支持方向键切换）、语音消息、弹幕，以及"对方正在看哪部视频"的在线状态提示。除了播放页内嵌的聊天面板，也有一个不依赖具体视频的独立聊天室页面。视频所在的文件夹里如果放了文档/压缩包/图片，列表页还会自动展示"相关附件"下载区和"图片库"，其中 Markdown 文档支持直接在线阅读，不用下载。

不开放注册，账号由管理员通过命令行脚本预先创建，密码使用 bcrypt 哈希存储，每个账号还可以配置一个自定义头像。

## 功能特性

- **登录鉴权**：用户名/密码存储在本地文件（bcrypt 哈希），不提供注册入口；登录失败次数过多会临时锁定；每个账号可以配置一个自定义头像，显示在聊天消息旁边
- **视频库**：
  - 递归扫描指定目录下的所有视频文件，支持多级文件夹浏览（带面包屑导航，可以点"根目录"或中间任意一级快速跳转）
  - 列表页展示自动生成的预览缩略图
  - **相关附件**：如果当前浏览的文件夹里直接放了文档（docx/doc/xlsx/xls/pptx/ppt/pdf/md/txt）或压缩包（zip/7z/rar），会在视频/文件夹网格下方自动出现"相关附件"栏目，每个文件显示对应类型的图标、文件名、大小，点击"下载"即可下载；其中 Markdown（.md）文件额外提供一个"阅读"按钮（在"下载"前面），点击直接在页面内弹层显示渲染后的全文——阅读弹层用的是比聊天气泡明显更大一档的排版（标题分级更醒目、引用块带底色、行距更宽松），照顾长文阅读体验，桌面端弹层宽度也比早期版本更宽；超过 2MB 的 .md 文件因为不适合整篇加载会提示改用下载；没有附件的话这个栏目不显示，不占用页面空间
  - **图片库**：如果当前文件夹里有图片（jpg/jpeg/png/gif/webp/bmp），会在"相关附件"上方显示"图片库"栏目，最多展示 8 张缩略图（缩略图尺寸和视频卡片一致）；点击任意一张会打开全屏 lightbox，支持左右/上下方向键切换、Esc 或点击空白处关闭，能翻遍这个文件夹下的**全部**图片（不只是首页展示出来的 8 张），底部会显示文件名和"第 N / 共 M 张"
  - 每个文件夹的浏览结果（子文件夹/视频/相关附件/图片库）都有一个很短的内存缓存（15 秒），减少来回切换文件夹时的重复磁盘扫描；这意味着刚放进去的新文件，最多可能要等 15 秒才会出现在页面上，不是绝对实时的。缓存过期后的扫描本身是异步的（不会同步阻塞其他人的视频播放/聊天等请求），且同一个目录如果被多个请求同时撞上缓存过期的瞬间，只会真正扫描一次、大家共用同一个结果，不会重复扫描
- **同步播放**：双方打开同一个视频会进入同一个"房间"，播放、暂停、拖动进度条都会实时同步给对方，并定期心跳纠正长时间播放后的进度漂移；同一个人开多个标签页进同一个房间也不会误判成"离开"；断线重连（或服务未重启期间双方都短暂离开）后会自动同步到房间最近一次的播放位置，不用从头开始（如果这次同步刚好发生在视频 metadata 还没加载完的时候，会等 metadata 就绪后再补上跳转，避免刚进页面时这一次同步偶发不生效）
- **实时聊天**：
  - 除了播放页内嵌的聊天面板，还有一个独立的聊天室页面（不依赖具体视频，随时可以进去和对方聊天），两边共用同一套消息渲染、表情、图片/语音发送逻辑
  - 支持完整的 Markdown 语法：
    - 行内格式：`**粗体**`、`*斜体*`、`_斜体_`（下划线斜体只在紧邻非字母数字/非 CJK 字符时才会触发，"user_name_field" 这类文件名/变量名/日期不会被误拆成斜体，中文/日文假名/韩文谚文都覆盖到了）、`~~删除线~~`、`` `代码` ``、行内数学公式 `$E=mc^2$`
    - 链接 `[文字](url)` 与图片 `![描述](url)`（只允许 http/https，链接额外允许 mailto，拒绝 `javascript:` 等危险协议；图片和下面说的上传图片一样支持点击全屏 lightbox 查看）
    - 标题 `# ~ ######`、分隔线 `---`
    - 多级引用 `>`、`>>`、`>>>`（支持任意层级嵌套）
    - 多级有序/无序列表（`-`、`*`、`+` 都可以作为无序列表符号），以及多级任务列表 `- [ ]` / `- [x]`（用缩进表示层级，2 个或 4 个空格都能正确识别层级，不需要严格对齐某一种缩进习惯）；列表项之间允许空行分隔（"松散列表"，每项内容会包一层 `<p>`），不会因为中间空了一行就被截断成两个列表
    - 表格，支持列对齐：`:---`（左）、`:---:`（居中）、`---:`（右）
    - 代码块（` ```语言名 ` 开头，会原样保留缩进/换行，且内部内容不会被其他 Markdown 语法误解析），右上角有一个"复制"按钮，点击直接复制代码原文到剪贴板
    - 块级数学公式 `$$...$$`
    - 折叠面板 `<details><summary>...</summary>...</details>`（仅识别这四个不带任何属性的精确标签，其余写法按纯文本显示，避免被用来夹带任意 HTML）
    - 裸 URL 自动识别为链接，URL 后面紧跟中文/日文假名/韩文谚文文字或标点（`https://x.com，继续说`、`https://x.comアクセス`）或西文标点（`https://x.com.` `(https://x.com)`）不会被一起吞进链接里
    - 弹幕里只渲染行内格式（不含图片、表格、代码块、折叠面板等块级结构，避免破坏弹幕飞行布局）
    - 数学公式渲染为带特殊样式的等宽/衬线字体文本，不引入 MathJax/KaTeX 等外部库，保持轻量；不支持复杂的 LaTeX 排版（分式、根号等符号本身不会被渲染成图形，只是原样显示公式源码）
  - 支持表情选择器
  - 支持发送图片（服务端会校验图片真实格式，不只是看文件扩展名），点击图片可以打开全屏 lightbox 查看，支持左右/上下方向键在当前聊天记录的所有图片间切换（不管是直接发送的图片，还是 Markdown 语法插入的行内图片），Esc 或点击空白处关闭
  - 支持发送语音消息：点击输入框前面的麦克风按钮开始录音，再点一次停止并自动发送（也可以点旁边的 ✕ 取消，不发送），最长录制 60 秒；消息气泡里是原生的音频播放条，可以直接拖动进度收听。为了保证语音在桌面浏览器和 iOS Safari 之间都能互相听到，服务端做了两件事：① 语音文件的访问路由支持 HTTP Range 请求（和视频流一样）——iOS Safari 播放 `<audio>` 前会先发一个探测性的 Range 请求，服务端如果不认 Range、只回完整文件，Safari 会直接拒绝播放（表现为播放条一直转圈）；② 如果服务器装了 ffmpeg（软依赖，见[环境要求](#环境要求)），上传后的语音会自动统一转码成 AAC/M4A——桌面 Chrome/Firefox 录音默认是 WebM/Opus，这个格式 iOS Safari 完全无法解码，不管 Range 支持得多好都放不出来，必须转成两边都认的格式才行；没装 ffmpeg 的话会退回直接使用录音原始格式，此时只有双方浏览器引擎兼容时才能互相听到（比如两边都用桌面 Chrome）。服务端同样只信任文件内容本身（不信任浏览器声称的格式），校验通过的语音文件和聊天图片共用同一套按天数自动清理的存储目录
  - 支持换行（Shift+Enter 换行，Enter 发送；用中文/日文/韩文输入法打字时，按 Enter 确认候选词不会被误判成"发送"，只有真正结束输入法组合状态后的 Enter 才会发送消息）
  - 支持弹幕：聊天内容可以选择以弹幕形式飘过视频画面，每条弹幕随机配色
  - 加入房间时会收到该房间最近的聊天记录（最多 50 条，存在内存里，重启服务会清空），刷新页面不会再"丢消息"；弱网抖动、手机切后台再切回来这类情况触发 Socket.IO 自动重连时，同样会重新收到一次历史记录（这是为了补齐断线期间错过的消息），客户端会按消息的时间戳自动过滤掉已经显示过的部分，不会看到聊天记录重复出现；长期没人用的房间，聊天记录和播放进度缓存会在空置超过 24 小时后自动清理，避免内存无限增长
  - 聊天消息、播放控制事件都有服务端限流（聊天最多 10 条/5 秒，播放控制最多 30 次/5 秒，按每个连接单独计算，同一个人开多个标签页/多个房间不会共用同一份额度而被误伤），防止异常脚本刷屏；服务端也会校验发送方确实在这个房间里，防止伪造消息/播放指令
- **在线状态**：
  - 视频列表页能看到对方"正在观看《XXX》/ 正在聊天中 / 浏览列表中 / 不在线"，并提供"加入TA观看"/"加入聊天"快捷按钮
  - 播放页、独立聊天室都能看到对方是否在同一个房间里，加入/离开都有实时提示
- **站点名称可自定义**：浏览器标签页标题、首页顶部品牌区显示的名字可以通过 `.env` 里的 `SITE_NAME` 配置，不设置则默认为"私密影院"
- **响应式设计**：PC 端和移动端均做了适配，移动端聊天面板可收起为抽屉，未读消息有数字提示
- **聊天面板宽度可调**：PC 端可以左右拖动视频区和聊天区之间的分隔线调整宽度（240px ~ 视口宽度的 60% 之间，且不超过 640px），宽度偏好会记住；双击分隔线恢复默认宽度
- **安全防护**：路径穿越防护、图片/语音上传文件签名校验、Session 防固定攻击、HttpOnly/SameSite Cookie 等（详见[安全设计](#安全设计)）

## 技术栈

- 后端：Node.js + Express 5 + Socket.IO 4
- 鉴权：express-session + bcryptjs
- 图片处理：multer（上传）+ ffmpeg（视频缩略图、聊天语音转码，均为可选）
- 前端：原生 HTML/CSS/JavaScript，无构建工具，无框架依赖

## 环境要求

- Node.js 18 及以上
- （可选）[ffmpeg](https://ffmpeg.org/)：用于生成视频预览缩略图，以及把聊天语音消息统一转码成跨平台兼容的格式（AAC/M4A）。没有安装也完全不影响登录、播放、聊天等核心功能——缩略图会显示默认的播放图标，语音消息则退回直接使用录音原始格式（同类浏览器引擎之间仍可互相听到，但跨生态比如"桌面 Chrome 录音 → iPhone 播放"可能放不出来）。

检查是否已安装 ffmpeg：

```bash
ffmpeg -version
```

没有的话，Ubuntu/Debian 系统可以这样安装：

```bash
sudo apt update && sudo apt install ffmpeg
```

## 安装步骤

### 1. 安装依赖

```bash
npm install
```

### 2. 配置环境变量

复制配置模板并编辑：

```bash
cp .env.example .env
```

打开 `.env`，至少需要修改这两项：

```bash
SESSION_SECRET=          # 用下面的命令生成一个随机值填进去
VIDEO_DIR=/真实存在的视频目录绝对路径
```

生成随机密钥：

```bash
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```

`.env` 完整字段说明：

| 字段 | 说明 |
|---|---|
| `PORT` | 服务监听端口，默认 3000 |
| `SITE_NAME` | 站点名称，显示在浏览器标签页标题、首页顶部品牌区。不设置则默认为"私密影院" |
| `SESSION_SECRET` | Session 签名密钥，必须是随机字符串且不少于 16 位 |
| `VIDEO_DIR` | 视频根目录的绝对路径，会递归扫描其下所有视频文件（支持多级子文件夹） |
| `NODE_ENV` | `development` 或 `production` |
| `TRUST_PROXY` | 是否信任反向代理传来的 `X-Forwarded-*` 头，默认 `false`。只有确实用了 nginx/Caddy 等反向代理时才设为 `true`，详见[用 Nginx 做反向代理](#用-nginx-做反向代理) |
| `CHAT_IMAGE_RETENTION_DAYS` | 聊天图片/语音保留天数，默认 30 天，超过的会被每天一次的定时任务自动清理；设为 `0` 关闭自动清理。如果一个聊天室一直没清空过、超过这个天数，房间历史记录里指向已被清理文件的图片/语音消息会自动换成一条"已过期"的文字提示，不会显示成加载失败的图片/转不动的语音 |

### 3. 创建账号

不提供注册页面，只能通过命令行脚本创建（建议正好创建两个账号）：

```bash
npm run create-user
```

按提示输入用户名和密码（密码输入时不会显示在终端上，且需要二次确认）。再运行一次即可创建第二个账号。

创建出来的账号默认头像是 `public/avatar/default.png`（会显示在聊天消息旁边）。想给某个账号换一个专属头像，把图片文件放进 `public/avatar/` 目录，再打开 `config/users.json`，把对应用户的 `avatar` 字段改成新文件的路径即可，格式大致如下：

```json
{
  "Alice": { "hash": "$2b$12$...", "avatar": "/avatar/default.png" },
  "Bob": { "hash": "$2b$12$...", "avatar": "/avatar/bob.png" }
}
```

`hash` 是 bcrypt 密码哈希，不要手动编辑；重新运行 `npm run create-user` 更新某个账号的密码时，会保留这个账号原有的 `avatar` 设置，不会被重置成默认头像。

### 4. 启动服务

```bash
npm start
```

看到下面的输出说明启动成功：

```
视频同步观影服务已启动: http://localhost:3000
视频目录: /your/video/dir
运行模式: development
```

浏览器打开 `http://localhost:3000`，用刚创建的账号登录即可。

### 5.（可选）放置 Logo 和网站图标

- Logo：把图片命名为 `logo.png`，放进 `public/images/logo.png`（建议高度 48-96px、透明背景 PNG）
- 网站图标（favicon）：把 `favicon.ico` 放在 `public/favicon.ico`（**注意是 `public/` 根目录下，不是放进 `images/` 文件夹里**），这样浏览器才能按标准约定自动找到它，所有页面统一生效

这两个文件不存在也不会报错，页面会自动隐藏 logo 位置、浏览器标签显示默认图标。


## 视频加载速度优化

如果觉得视频播放启动得比较慢（尤其是码率明明不高、但点开要等好几秒才开始播放），最常见的原因是 **MP4 文件的索引信息（moov atom）被放在了文件末尾**——浏览器需要先摸到文件尾部才能开始播放，文件越大这个延迟越明显。这常见于手机直接录制、某些录屏/转码工具导出的视频。

可以用项目自带的小工具做无损修复（只重新排列容器结构，不重新编码画面/声音，速度很快）：

```bash
node scripts/optimizeVideo.js 你的视频.mp4
```

会在同目录生成 `你的视频.faststart.mp4`，确认播放没问题后，自行决定是否替换原文件。这个工具需要 ffmpeg。

## 安全设计

| 风险点 | 防护方式 |
|---|---|
| 密码泄露 | bcrypt 哈希存储，永不存明文；校验时无论用户名是否存在都会执行一次哈希比对，降低时序侧信道风险 |
| 路径穿越 | 视频文件、聊天图片/语音、相关附件、图片库里的每一个文件真实路径都会校验确实落在指定根目录内，拒绝 `../` 等穿越尝试；相关附件/图片库还会额外校验扩展名，不允许下载或查看白名单之外的文件类型；Markdown 在线阅读接口在此基础上额外限定只能是 `.md` 文件 |
| 伪造图片/语音上传 | 不信任客户端声称的文件扩展名/Content-Type，读取文件头"魔数"校验真实格式（图片校验 PNG/JPG/GIF/WEBP 文件头，语音校验 WebM/Ogg/MP4 容器头）；上传文件名由服务端随机生成，不使用用户提供的文件名 |
| 未授权访问 | 所有路由（含视频流、缩略图、聊天图片/语音、相关附件下载与在线阅读、图片库查看、WebSocket 连接）都会校验登录状态 |
| 伪造播放/聊天消息 | Socket.IO 服务端会校验发送方确实通过 join-room 进过对应的房间，不在房间里的连接发来的聊天消息/播放控制指令会被直接丢弃，不会被广播或写入历史记录 |
| 暴力破解登录 | 同一 IP+用户名连续失败 5 次后锁定 15 分钟 |
| Session 劫持/固定 | Cookie 设置 `HttpOnly`、`SameSite=Lax`；登录成功后重新生成 Session ID |
| 聊天 XSS | 聊天消息的 Markdown 渲染先转义全部 HTML 特殊字符，再叠加自己可控的安全标签；链接/图片只允许 `http`/`https`（链接额外允许 `mailto`），拒绝 `javascript:` 等危险协议；折叠面板 `<details>`/`<summary>` 只精确识别这 4 个不带任何属性的标签字符串，不支持任意 HTML |
| 刷屏/恶意消耗 | 聊天消息（10 条/5 秒）和播放控制事件（30 次/5 秒）都有服务端限流，按每个连接单独计算，超出静默丢弃 |
| 跨站点滥用 | 响应头设置 `X-Content-Type-Options: nosniff`、`X-Frame-Options: DENY`、`Referrer-Policy: same-origin`，以及 `Content-Security-Policy`（限制脚本只能加载同源文件，禁止内联脚本执行，作为聊天 XSS 防护失效时的最后一道兜底） |

## 部署到公网时的建议

1. **务必上 HTTPS**：用 Nginx/Caddy 做反向代理 + Let's Encrypt 证书。否则密码和视频流都是裸传输的。
2. **不建议直接对公网开放端口**：这个网站设计上只给两个人用，更推荐用 [Tailscale](https://tailscale.com)、WireGuard 之类的工具组个私人内网，根本不暴露公网端口，比加固一个公网服务安全得多。
3. 如果一定要公网部署，建议加防火墙 IP 白名单，并用 `pm2` 或 `systemd` 管理进程，保证崩溃后能自动重启。

### 用 Nginx 做反向代理

项目根目录的 `nginx.conf.example` 是一份可以直接参考的配置文件，包含两个容易被忽略但很关键的点：

- **WebSocket 升级头**（`Upgrade` / `Connection: upgrade`）：少了这两行，Socket.IO 连不上，播放同步、聊天、在线状态这些实时功能全部失效（但页面本身还是能打开的，容易让人摸不着头脑）。
- **`X-Forwarded-Proto` 等头**：要配合 `.env` 里的 `TRUST_PROXY=true` 一起生效，否则即使配了也不会被信任，见下方说明。

**用了 Nginx 反代之后，记得在 `.env` 里把 `TRUST_PROXY` 设为 `true`：**

```bash
TRUST_PROXY=true
```

这一项默认是 `false`，因为只有真的有反向代理挡在前面时才能开启——否则客户端能直接伪造 `X-Forwarded-*` 头，冒充任意 IP 或冒充 HTTPS 连接，是个安全隐患。具体原理见下面"常见问题"里的说明。

## 常见问题

**Q: 配置了 `NODE_ENV=production` 并接上 Nginx 反代后，输入用户名密码无法跳转到首页？**
A: 几乎可以肯定是 Cookie 的 `Secure` 属性导致的——浏览器规则规定，标记为 `Secure` 的 Cookie 只能在 HTTPS 连接下被接受。如果 Nginx 暂时还没配 HTTPS（只是普通 HTTP 反代），登录请求本身在服务端是成功的，但浏览器会直接拒绝存储这个登录态 Cookie，于是跳转到首页时又被当成"未登录"打回登录页。

解决方法：在 `.env` 里加上 `TRUST_PROXY=true` 并重启服务（同时确保 Nginx 配置里带上了 `X-Forwarded-Proto` 头，参考 `nginx.conf.example`）。加上这一项后，Cookie 是否带 `Secure` 标记会根据实际连接是否为 HTTPS 动态判断，而不是简单粗暴地跟 `NODE_ENV` 绑定——这样不管你的 Nginx 当前是 HTTP 还是 HTTPS，登录都能正常跳转，同时一旦升级到 HTTPS，安全性会自动跟着提升。

**Q: 视频列表里没有显示缩略图，只看到默认的播放图标？**
A: 服务器没装 ffmpeg，或者该视频文件无法被 ffmpeg 正常解码。这不影响播放和其他功能，装好 ffmpeg 重启服务即可（参考[环境要求](#环境要求)）。

**Q: "相关附件""图片库"支持哪些文件格式？某个文件夹放了图片/文档，但栏目没有出现？**
A: "相关附件"支持 docx/doc/xlsx/xls/pptx/ppt/pdf/md/txt 和 zip/7z/rar，其中 .md 文件除了"下载"还会额外出现一个"阅读"按钮，点击直接在页面内弹层显示渲染后的全文（超过 2MB 的 .md 文件因为不适合整篇加载到内存/页面里，会提示改用下载）；"图片库"支持 jpg/jpeg/png/gif/webp/bmp（不支持 svg——SVG 文件可以嵌入脚本，直接打开有被浏览器执行的风险，所以整体排除掉了）。两个栏目都只扫描"当前浏览的这一层文件夹"，不会递归查子文件夹里的文件，如果文件放在子文件夹里，需要点进那个子文件夹才能看到对应栏目。

**Q: 点击麦克风按钮提示"录音功能需要通过 HTTPS（或 localhost）访问"，或者提示"无法访问麦克风"？**
A: 前一种提示是浏览器的安全限制：`getUserMedia`（录音需要用到的浏览器 API）只在"安全上下文"下可用，也就是 `https://` 或者 `http://localhost`，普通的 `http://` 局域网/公网地址不行——这不是"浏览器不支持录音"，而是访问方式本身不满足安全要求，所以不管换什么浏览器、换 PC 还是手机测试都会一样失败。如果你是通过 IP 或域名以 HTTP 方式直接访问（没有走 HTTPS 反代），需要参考[部署到公网时的建议](#部署到公网时的建议)给 Nginx 配好 HTTPS 才能用上录音功能。如果已经是 HTTPS 访问但提示"无法访问麦克风"，检查一下浏览器的网站权限设置里是否手动拒绝过麦克风权限。

**Q: Markdown 阅读/聊天消息里，Markdown 排版和 [Editor.md](https://pandao.github.io/editor.md/) 之类成熟编辑器比还是有差距（没有代码高亮、脚注、KaTeX 数学公式实际渲染、标题锚点等）？**
A: 这是真实的已知局限：项目里的 Markdown 渲染是不依赖任何第三方库、手写的一个轻量解析器（标题/列表/引用/表格/代码块/任务列表/行内格式都支持，也修过"列表遇空行截断""4 空格缩进层级错位"这类兼容性问题），但确实没有实现代码语法高亮、脚注、数学公式实际渲染成图形、标题锚点跳转这些更进阶的功能。如果想要完整的 Editor.md 级别体验，需要引入 [marked.js](https://marked.js.org/) + [DOMPurify](https://github.com/cure53/DOMPurify)（解析 + 输出净化）等成熟库替换掉现在的手写实现——这是一个已知的可选改进方向，只是本项目出于"保持零依赖、体积小"的取舍暂时没有引入，需要的话可以自行把这两个库的 `.min.js` 放进 `public/js/lib/` 再对接。

**Q: 移动端点击播放后视频自动全屏，看不到聊天了？**
A: 这是部分移动浏览器（尤其 iOS Safari）的默认行为，项目里已经给 `<video>` 标签加了 `playsinline` 相关属性来阻止这个行为。如果仍有问题，请检查浏览器版本，或反馈具体机型/浏览器。

**Q: 忘记密码怎么办？**
A: 没有"忘记密码"流程（设计上就不提供注册和找回，降低攻击面）。重新运行 `npm run create-user`，输入同一个用户名即可覆盖更新密码。

**Q: 想加更多账号，能超过两个人吗？**
A: 技术上可以（`npm run create-user` 可以一直创建新账号），但同步播放和聊天的房间设计是按"视频 ID"分房间的，超过两人进入同一个视频会一起同步、一起聊天，没有针对多人场景做特别优化（比如没有限制房间人数）。

**Q: 服务重启后两个人都要重新登录，为什么不把 Session 存到数据库里？**
A: 这是有意的取舍。Session 目前存在内存里，服务重启/崩溃会清空，需要重新登录——但对只有两个人的私用场景来说，重新登录就是输一遍密码，几秒钟的事，引入数据库/文件持久化带来的额外依赖和复杂度并不值得。

**Q: 视频流要不要做限速，防止占满带宽？**
A: 没有做。这是为多人共享、互不信任场景设计的防护（防止一个用户占满所有人的带宽），但这个项目设计上就是给两个互相信任的人私用的，限速只会让正常观看体验变差（可能引发卡顿），收益和成本不成比例，所以没加。如果你确实需要限速（比如服务器本身带宽很有限），更合适的做法是在 Nginx 反代层用 `limit_rate` 指令控制，而不是在应用层处理。
